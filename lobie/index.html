How it works
From SQL to on-chain in 4 steps
The protocol does ONE thing: publish audited per-unit cash flows. Every economic layer is built on top.
Step 01
SQL
Source Data
Lobie's property management system records monthly revenue, expenses, and owner payouts for each unit in a GCP SQL database.
Step 02
NFT
Mint Unit
Each unit gets one ERC-721 NFT with a deterministic tokenId derived from building + unit code. No PII on-chain (LGPD compliant).
Step 03
R$
Post Observations
Monthly financial data posted on-chain: gross revenue, 11 expense categories, net result, owner payout. Each observation has a canonical row hash.
Step 04
HT
Auditor Attests
Harris & Trotter reconciles on-chain data against bank movements, then attaches an EAS attestation UID to each verified observation.
Architecture
From property to DeFi primitive
The data pipeline that turns private SQL into composable on-chain valuations.
On-Chain Data
Full expense decomposition per unit
Every monthly observation contains 11 expense categories plus revenue and payout — the same granularity as the internal accounting system.
R$
Revenue & Payout
grossRevenue (valor_bruto_reservas), netResult (resultado_liquido), and ownerPayout (repasse_unidade) — the valuation-relevant line.
11
Expense Categories
OTA fees, taxes, cleaning, Lobie fee, condominio, energia, IPTU, manutencao, enxoval, and other expenses — all on-chain.
NPV
Gordon Growth Valuation
npvOf(tokenId, r, g) computes on-chain NPV using trailing 12-month average payout. Default: r=6%, g=3%. Any protocol can call it.
#
Canonical Row Hash
Each observation includes keccak256 of the canonical row (excluding PII). Auditors verify the hash matches source data — full audit trail.
EAS
Auditor Attestation
Harris & Trotter attaches EAS attestation UIDs after bank reconciliation. Corrections reset the attestation — HT must re-attest.
0
Zero PII On-Chain
No names, emails, CPFs, or personal data ever touch the blockchain. LGPD compliant by design. Only financial data and hashes.
